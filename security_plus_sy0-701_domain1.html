1.1 Compare and contrast various types of security controls

Categories of Controls (WHO/HOW it's implemented)

  • Technical Implemented through technology (firewalls, IDS, encryption, antivirus, ACLs).
  • Managerial Administrative — policies, risk assessments, security planning, training programs.
  • Operational Day-to-day procedures done by people (awareness training, incident response, backups, user provisioning).
  • Physical Tangible barriers (locks, fences, guards, cameras, bollards).

1.2 Fundamental Security Concepts

CIA Triad

  • Confidentiality — only authorized parties can read data (encryption, access controls).
  • Integrity — data is not altered improperly (hashing, digital signatures).
  • Availability — data/services are accessible when needed (redundancy, backups, DDoS mitigation).

Non-Repudiation

Ensures a party cannot deny an action they performed. Achieved via digital signatures, audit logs, and video evidence.

AAA Framework

  • Authentication — proves identity (something you know/have/are/do/somewhere you are).
  • Authorization — defines what an authenticated subject may do (RBAC, ABAC, MAC, DAC).
  • Accounting — tracks/records actions for audit (logs, RADIUS/TACACS+).
  • Authentication applies to people AND to systems (e.g., device certificates, mutual TLS).

Gap Analysis

Comparison of current security posture vs. desired/required state (e.g., NIST CSF, ISO 27001) to identify what's missing.

Zero Trust — "Never trust, always verify"

No implicit trust based on network location. Every request is authenticated, authorized, and continuously validated.

PlaneComponents
Control Plane • Adaptive Identity — context-aware (location, device, risk)
• Threat Scope Reduction — minimize attack surface
• Policy-Driven Access Control — decisions from defined policies
• Policy Engine (PE) — makes the access decision
• Policy Administrator (PA) — establishes/communicates the decision
Data Plane • Implicit Trust Zones — areas where trust is assumed (Zero Trust eliminates these)
• Subject/System — the user or device requesting access
• Policy Enforcement Point (PEP) — enforces the PA's decision (gateway/proxy)

Physical Security

  • Bollards — short posts blocking vehicle ramming.
  • Access control vestibule (mantrap) — two-door system preventing tailgating.
  • Fencing — perimeter deterrent/barrier.
  • Video surveillance (CCTV) — detective + deterrent.
  • Security guard — flexible, judgment-based control.
  • Access badge — RFID/smartcard for identification + access.
  • Lighting — deterrent and aids surveillance.
  • Sensors:
    • Infrared — detects heat/body movement
    • Pressure — triggers when weight applied
    • Microwave — detects motion via reflected microwaves
    • Ultrasonic — detects motion via sound-wave reflection

Deception and Disruption Technology

  • Honeypot — decoy system to attract/study attackers.
  • Honeynet — entire decoy network of honeypots.
  • Honeyfile — fake file (e.g., "passwords.xlsx") that triggers an alert if accessed.
  • Honeytoken — fake data/credential planted as a tripwire (e.g., fake API key).

1.3 Change Management Processes & Security Impact

Business Processes

  • Approval process — formal sign-off (often via Change Advisory Board / CAB).
  • Ownership — who is accountable for the change.
  • Stakeholders — anyone affected (users, ops, security, business owners).
  • Impact analysis — what could break or be affected (security, compliance, uptime).
  • Test results — proof the change works in test/staging.
  • Backout plan — how to roll back if the change fails.
  • Maintenance window — agreed time slot to perform the change.
  • Standard operating procedure (SOP) — documented steps for repeatable changes.

Technical Implications

  • Allow lists / Deny lists — updates to firewall, EDR, app controls.
  • Restricted activities — what's not permitted during the window.
  • Downtime — planned outage of services.
  • Service / Application restart — required to apply changes.
  • Legacy applications — limited support; risky to change.
  • Dependencies — other systems affected by the change.

Documentation

  • Update diagrams (network, data flow).
  • Update policies and procedures.

Version Control

Tracks changes to code, configs, and documents (e.g., Git). Enables rollback, accountability, and audit.

1.4 Cryptographic Solutions

Public Key Infrastructure (PKI)

  • Public key — shared openly; encrypts data or verifies signatures.
  • Private key — kept secret; decrypts or signs.
  • Key escrow — third party holds a copy of keys for recovery / legal access.

Encryption

Levels: Full-disk, Partition, File, Volume, Database, Record.
Transport / communication: TLS, IPsec, SSH protect data in motion.

SymmetricAsymmetric
KeysOne shared secretPublic + private pair
SpeedFastSlow
UseBulk data encryptionKey exchange, signatures
ExamplesAES, ChaCha20, 3DESRSA, ECC, DSA, Diffie-Hellman
  • Key exchange — securely sharing keys (e.g., Diffie-Hellman, ECDH).
  • Algorithms — choose modern/strong (AES-256, RSA-2048+, ECC).
  • Key length — longer = stronger (subject to algorithm).

Cryptographic Tools

  • TPM (Trusted Platform Module) — chip on motherboard storing keys, performing crypto, supports BitLocker / secure boot.
  • HSM (Hardware Security Module) — dedicated appliance for high-volume key management (CAs, payment systems).
  • Key Management System (KMS) — centralized service to generate/store/rotate keys (often cloud-based: AWS KMS, Azure Key Vault).
  • Secure Enclave — isolated CPU region for sensitive ops (Apple Secure Enclave, Intel SGX).

Obfuscation

  • Steganography — hiding data inside other data (e.g., image, audio).
  • Tokenization — replacing sensitive data with a non-sensitive token (common in PCI/credit cards). Reversible via lookup.
  • Data masking — partial hiding (e.g., XXXX-XXXX-XXXX-1234). Often irreversible / for display.

Hashing & Salting

  • Hashing — one-way function producing fixed-length digest (SHA-256, SHA-3). Used for integrity, password storage.
  • Salting — random value added before hashing to defeat rainbow tables.
  • Key stretching — repeated hashing (PBKDF2, bcrypt, scrypt, Argon2) to slow brute-force.

Digital Signatures

Sender hashes a message and encrypts the hash with their private key. Recipient decrypts with sender's public key and compares hashes. Provides integrity, authentication, non-repudiation.

Blockchain & Open Public Ledger

Distributed, immutable ledger where each block is hash-linked to the previous. Anyone can verify; tampering breaks the chain. Use cases: cryptocurrency, supply chain, identity.

Certificates

  • Certificate Authority (CA) — trusted issuer that signs certificates.
  • CRL (Certificate Revocation List) — list of revoked certificates published by the CA.
  • OCSP (Online Certificate Status Protocol) — real-time check of cert revocation status (faster than CRL).
  • Self-signed — signed by the same entity it identifies; used internally / not trusted by browsers.
  • Third-party — issued by a public CA (DigiCert, Let's Encrypt) — trusted by browsers.
  • Root of trust — top of the cert chain; preinstalled trusted root CAs.
  • CSR (Certificate Signing Request) — file generated containing public key + identity info, sent to CA for signing.
  • Wildcard certificate — covers all subdomains of a domain (*.example.com).
