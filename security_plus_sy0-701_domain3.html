3.1 Architecture Models & Their Security Implications

Cloud

  • Responsibility Matrix — divides security responsibility between provider and customer:
    LayerIaaSPaaSSaaS
    Data & AccessCustomerCustomerCustomer
    App / RuntimeCustomerProviderProvider
    OS / HypervisorProviderProviderProvider
    HardwareProviderProviderProvider
  • Hybrid — mix of on-prem + cloud; complex identity, networking, and data flow.
  • Third-party vendors — SaaS, MSPs — evaluate via due diligence, contracts, audits (SOC 2).

Other Architecture Concepts

  • Infrastructure as Code (IaC) — managing infra with code (Terraform, CloudFormation). Versioned, testable, repeatable; risk = secrets in code, drift.
  • Serverless — code runs in functions (Lambda); no server management. Reduces patching burden but introduces opaque event-driven attack surface.
  • Microservices — many small services communicating via APIs. Better isolation but more network-exposed APIs to defend.

Network Infrastructure

  • Physical isolation — separate equipment/cabling. Air-gapped: no network connection at all (used for OT / classified).
  • Logical segmentation — VLANs, subnets, firewall zones.
  • SDN (Software-Defined Networking) — separates control plane from data plane; dynamic, programmatic network configuration.

Other Models

  • On-premises — full control + full responsibility; high CapEx.
  • Centralized vs. Decentralized — central is easier to secure/manage; decentralized is more resilient but harder to govern.
  • Containerization (Docker, Kubernetes) — lightweight, share host kernel; faster than VMs but weaker isolation.
  • Virtualization — hypervisor isolates VMs; stronger boundary, more overhead than containers.
  • IoT — limited compute, often unpatched; segregate on dedicated network.
  • ICS / SCADA — industrial control; legacy protocols (Modbus), uptime over patching, physical-world consequences.
  • RTOS (Real-Time OS) — deterministic timing; often in embedded medical/defense/automotive.
  • Embedded systems — purpose-built devices; difficult to patch.
  • High Availability (HA) — designed to keep services running during failures.

Architecture Considerations

When selecting an architecture, evaluate: Availability, Resilience, Cost, Responsiveness, Scalability, Ease of deployment, Risk transference (e.g., to a cloud provider), Ease of recovery, Patch availability, Inability to patch (legacy/OT), Power, Compute.

3.2 Securing Enterprise Infrastructure

Infrastructure Considerations

  • Device placement — where in the topology a security tool sits (perimeter vs. internal).
  • Security zones — DMZ, internal, restricted; firewalls between zones.
  • Attack surface — sum of all exposure points; minimize by hardening, disabling services.
  • Connectivity — wired/wireless, VPN, dedicated lines.
  • Failure modes:
    • Fail-open — on failure the device passes traffic (availability over security).
    • Fail-closed — on failure the device blocks traffic (security over availability).
  • Device attributes:
    • Active vs. Passive — active blocks (IPS); passive only observes (IDS).
    • Inline vs. Tap/Monitor — inline is in the data path; tap/monitor receives a copy via SPAN port.

Network Appliances

ApplianceRole
Jump ServerHardened bastion host used to reach restricted zones; admins SSH/RDP through it.
Proxy ServerForwards requests on behalf of clients; can filter/log/cache. Forward proxy = client-side; reverse proxy = server-side.
IDSDetects intrusions and alerts (passive).
IPSDetects AND blocks intrusions (active, inline).
Load BalancerDistributes traffic across servers — improves availability + scalability.
SensorsCollect telemetry (network/host) for SIEM/IDS.

Port Security

  • 802.1X — port-based network access control. Supplicant → Authenticator (switch/AP) → Authentication Server (RADIUS).
  • EAP — extensible authentication protocol used over 802.1X (EAP-TLS, PEAP, EAP-TTLS, EAP-FAST).

Firewall Types

  • Layer 4 firewall — stateful packet filter (IP, port, protocol).
  • Layer 7 firewall — application-aware (deep packet inspection, app identification).
  • NGFW (Next-Generation Firewall) — Layer 4–7 + IPS, app-ID, user-ID, decryption.
  • WAF (Web Application Firewall) — protects web apps from SQLi, XSS, etc.
  • UTM (Unified Threat Management) — single appliance combining firewall, IPS, AV, content filter, VPN, etc.

Secure Communication / Access

  • VPN — encrypted tunnel over an untrusted network; site-to-site or remote-access.
  • Tunneling — TLS — common for SSL/TLS VPNs (clientless via browser, or full client).
  • Tunneling — IPsec — operates at Layer 3; AH (integrity), ESP (encryption + integrity); Transport (payload only) vs. Tunnel (whole packet); uses IKE for key exchange.
  • SD-WAN — software-defined WAN; intelligently routes traffic across multiple links (MPLS, broadband, LTE) for cost/performance.
  • SASE (Secure Access Service Edge) — converges SD-WAN + cloud-delivered security (SWG, CASB, ZTNA, FWaaS) into one service.

Selection of Effective Controls

Pick controls based on threat model, regulatory requirements, cost, and operational impact. Defense-in-depth = layered controls so no single failure breaches everything.

3.3 Data Protection Concepts & Strategies

Data Types

  • Regulated — covered by laws (PII, PHI, PCI).
  • Trade secret — confidential business info (formula, process).
  • Intellectual Property (IP) — copyrights, patents, trademarks.
  • Legal — contracts, litigation hold materials.
  • Financial — bank, accounting, payroll data.
  • Human-readable — text, documents people can read directly.
  • Non-human-readable — binary, encrypted, machine code.

Data Classifications

LevelDescription
PublicNo harm if disclosed (marketing material).
PrivateInternal use; minor harm if leaked.
SensitiveLoss could harm individuals/business.
ConfidentialRestricted to specific groups; serious damage if exposed.
RestrictedHighly limited access; significant harm if disclosed.
CriticalHighest impact level; loss is catastrophic.

Data States

  • Data at rest — stored (disk, DB, backup tape). Protect with encryption (BitLocker, TDE).
  • Data in transit (in motion) — moving on a network. Protect with TLS, IPsec, SSH, VPN.
  • Data in use — being processed in memory. Hardest to protect; use enclaves (SGX), homomorphic encryption, confidential computing.

Sovereignty & Geolocation

  • Data sovereignty — data is subject to the laws of the country where it's stored.
  • Geolocation — knowing where data physically resides; often required by GDPR / regional laws.

Methods to Secure Data

  • Geographic restrictions — block storage/access in certain regions (geofencing).
  • Encryption — protects at rest and in transit.
  • Hashing — integrity check; one-way.
  • Masking — partial obscuring (XXXX-1234) for display.
  • Tokenization — replace with non-sensitive token (PCI-friendly).
  • Obfuscation — make data harder to read (steg, code obf).
  • Segmentation — separate sensitive data into isolated networks/databases.
  • Permission restrictions — RBAC/ABAC, NTFS ACLs, S3 policies.

3.4 Resilience & Recovery in Security Architecture

High Availability (HA)

  • Load balancing — distributes traffic across multiple active servers (active/active).
  • Clustering — multiple servers act as a single logical unit; can be active/active or active/passive (failover).

Site Considerations (DR)

SiteEquipmentDataRecovery timeCost
HotFully equippedLive/replicatedMinutes–hoursHighest
WarmEquipped, partial dataPeriodic syncHours–daysMedium
ColdEmpty space + power/connectivityNone until restoredDays–weeksLowest
  • Geographic dispersion — distribute sites across regions to survive disasters affecting an entire area.

Diversity & Cloud

  • Platform diversity — using different vendors/OSes so a single vuln doesn't take everything down.
  • Multi-cloud — using AWS + Azure + GCP for resilience and to avoid lock-in.
  • Continuity of operations (COOP) — keeping critical functions running through disruptions; covered in BCP.

Capacity Planning

  • People — staffing levels, on-call rotation, cross-training.
  • Technology — software licenses, automation capacity.
  • Infrastructure — compute, storage, network, power, cooling.

Testing

  • Tabletop exercise — discussion-based walkthrough of a scenario.
  • Failover test — actually switch to backup systems to verify they work.
  • Simulation — execute responses in a controlled mock incident.
  • Parallel processing — run primary and backup at the same time to compare outputs.

Backups

  • Onsite vs. offsite — onsite is fast; offsite survives site disasters. Best practice = both (3-2-1 rule: 3 copies, 2 media types, 1 offsite).
  • Frequency — driven by RPO (Recovery Point Objective).
  • Encryption — backups are a high-value target; always encrypt.
  • Snapshots — point-in-time copy (often VM/disk-level); fast restore.
  • Recovery — verify restores actually work — untested backups are not backups.
  • Replication — continuous copying to another location (sync = real-time, async = delayed).
  • Journaling — log of every change so the system can replay/roll back transactions (DBs, file systems).

Power

  • UPS (Uninterruptible Power Supply) — battery providing short-term ride-through during outages and clean transition to generator.
  • Generator — long-term backup power (diesel/natural gas) for extended outages.
